## 2. 各目录写入规则

| 目录 | 写入规则 |
|------|---------|
| `00-me/` | 只写入用户个人的决策信息（定位、产品、客户、账号、目标）。内容为用户手动填写的配置，不生成自动化内容。文件名不加 emoji 前缀。 |
| `10-raw/` | 只写入原始素材（别人写的东西、截图、链接）。不修改已有文件内容。默认写入 `inbox/` 子目录。 |
| `20-wiki/` | 只写入用户消化后的**结论性知识**。必须是用自己的话总结的，不是复制粘贴。文件名不加 emoji 前缀。 |
| `30-flow/` | 写入业务流程产出（选题卡、文章草稿、发布内容、CRM记录等）。内容文件需要 emoji 状态前缀。此处不写知识，只调用 wiki 中的知识。 |
| `40-agents/` | 写入 AI 相关的 Prompt、配置、使用记录。不写业务知识。 |
| `50-projects/` | 写入跨流程的实体资料（客户档案、品牌规范、账号详细资料）。不加 emoji 前缀。 |
| `60-assets/` | 只写入已经验证过、可直接复用的成品素材。半成品放 flow，成品放这里。 |
| `99-archive/` | 只写入已归档的文件。不要把还在使用的内容放这里。 |

## 3. 文件命名铁律

### 内容文件（有状态流转的）
```
格式：{emoji状态}-{YYYY-MM-DD}-{标题}.md
正则：^[📋📝🎥✂️📤✅🚫]-\d{4}-\d{2}-\d{2}-.+\.md$
```

示例：
- `📋待选-2026-05-05-如何建立个人IP.md`
- `📝草稿-2026-05-05-小红书爆款标题公式.md`
- `✅已发布-2026-05-01-3个获客技巧.md`

### 系统文件（00-me、wiki、projects、assets 下的文件）
```
格式：{标题}.md
```
不加 emoji 前缀，不加日期前缀。

### 模板文件
```
格式：_template-{用途}.md
```
以下划线开头，表示模板。

### 禁止事项
- 禁止在文件名中使用中文日期（必须用 YYYY-MM-DD）
- 禁止在文件名中使用空格（用连字符 - 替代）
- 禁止在系统文件（00-me、wiki、projects、assets）上加 emoji 状态前缀
- 禁止在 raw 目录下修改已有文件的内容
- 禁止在 wiki 目录下放入未经消化的原始资料

## 4. 状态管理规则

### 状态流转（必须按此顺序）
```
图文/文章：📋待选 → 📝草稿 → 📤待发布 → ✅已发布 → 🚫Closed
视    频：📋待选 → 📝草稿 → 🎥拍摄中 → ✂️剪辑中 → 📤待发布 → ✅已发布 → 🚫Closed
```

### 双标记机制
每次变更状态时，必须同时更新两处：

1. **文件名 emoji 前缀** — 重命名文件，替换开头的 emoji
2. **frontmatter status 字段** — 更新 `status:` 的值

```yaml
---
title: 如何建立个人IP
status: 草稿        # 必须与文件名 emoji 对应
type: 文章
platform: 公众号
created: 2026-05-05
updated: 2026-05-05
tags: [个人IP, 自媒体, 获客]
---
```

### status 字段与 emoji 的对应关系
| emoji | status 值 |
|-------|----------|
| 📋 | 待选 |
| 📝 | 草稿 |
| 🎥 | 拍摄中 |
| ✂️ | 剪辑中 |
| 📤 | 待发布 |
| ✅ | 已发布 |
| 🚫 | Closed |

### 改状态时的操作
1. 将文件名中的 emoji 替换为新状态的 emoji
2. 更新 frontmatter 中的 `status` 字段
3. 更新 frontmatter 中的 `updated` 日期为当天
4. **不要移动文件** — 文件从生到死不搬家，状态变更只改名字和 frontmatter

## 7. 禁止操作

1. **禁止修改 raw 目录下的已有文件内容** — raw 只进不改
2. **禁止在 wiki 目录下放入未消化的原始资料** — wiki 只放结论
3. **禁止在 flow 目录下生产知识性内容** — flow 只做调用
4. **禁止移动文件来变更状态** — 状态变更只改文件名和 frontmatter
5. **禁止跳过状态流转** — 必须按 📋→📝→📤→✅→🚫 的顺序推进（视频含 🎥→✂️）
6. **禁止在文件名中使用空格** — 用连字符 `-`
7. **禁止给系统文件（00-me/wiki/projects/assets）加 emoji 状态前缀**
8. **禁止在 frontmatter 中使用与文件名不一致的 status 值**
9. **禁止将还在使用的内容移到 99-archive/**
10. **禁止创建不在上述目录结构中的新一级目录** — 如需扩展，在现有目录下创建子目录
11. **禁止移动文件来变更状态或归档** — 文件从生到死不搬家，状态变更只改文件名和 frontmatter

## 8. 引导填写模式（三段式模板）

当被要求创建一个需要用户填写的文件（尤其是 00-me 下的文件）时，必须使用三段式格式：

```markdown
# [文件标题]

## 📌 填写指南
<!-- 告诉用户：这个文件是什么、为什么要填、怎么填、填完有什么用 -->
<!-- 用2-3段通俗的话解释清楚 -->

## ✏️ 你的内容
<!-- 用户实际填写的区域 -->
<!-- 如果是配置类文件，用填空式引导 -->

## 📖 参考：[行业]示例
<!-- 提供一个具体行业的完整填写示例 -->
<!-- 让用户有明确的参照，降低填写门槛 -->
```

### 三段式原则
- **📌 填写指南**：用大白话解释，不超3段，不说教
- **✏️ 你的内容**：用填空式引导，每个填空项都有简短提示
- **📖 参考**：提供一个真实可用的行业示例，用户可以直接模仿
