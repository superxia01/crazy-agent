# Vault 骨架设计说明（v4）

## 核心架构：底层优先

```
00-me/        # 🧠 决策层：我是谁、我卖什么、卖给谁
10-raw/       # 📥 输入层（只进不改）
20-wiki/      # 📚 知识层（只放结论）
30-flow/      # 🔁 业务流程层（只做调用，不生产知识）
40-agents/    # 🤖 AI能力层（统一）
50-projects/  # 📦 实体层（跨流程的业务实体）
60-assets/    # 💰 可复用资产
99-archive/   # 🗄️ 归档
```

## 底层优先架构

v4 采用底层优先架构，将 me/raw/wiki 作为独立顶级目录，而非嵌套在每个流程目录中。

设计理由：
1. 素材天然多用途，不应在收集阶段预分配到某个流程
2. raw 是统一收件箱，拆碎后变成三个孤岛
3. me 是配置中心，按流程打散后找东西变难
4. 用户最怕的不是找不到东西，而是"不知道该放哪所以干脆不整理"

## 三层铁律

**raw = 只进不改**
素材仓库，不做整理。扔进去就行。

**wiki = 只放结论**
不是资料，是你消化后的东西。判断标准：如果你用自己的话重新写不出来，说明还没消化，放回 raw。

**flow = 只做调用**
不生产知识，只用知识。写文章时去 wiki 找痛点、找方法论，在 flow 里完成创作。

## 整体数据流

```
            00-me（操作系统：我是谁、卖给谁、怎么干）
              ↓ 定义
10-raw → 20-wiki → 30-flow → 50-projects
 只进不改   只放结论   只做调用    落地实体
              ↑           ↑
         40-agents 辅助全链路
         60-assets 提供可复用素材
```

## 各层职责

### 00-me/ — 决策层
回答"我是谁、我卖什么、卖给谁"。所有需要用户个人决策的内容。
- 10-定位与品牌：IP定位、品牌风格、个人故事
- 20-产品与业务：产品清单、价格体系、交付标准
- 30-目标客户：客户画像、客户分层
- 40-账号矩阵：各平台账号信息
- 50-目标与节奏：商业目标、日常工作节奏

### 10-raw/ — 输入层
所有素材的唯一入口。不整理、不分类、不做判断。
- inbox/：扔进去就行，不看第二遍
- cleaned/：去广告、排好版，但还是别人的东西
- tagged/：加了标签方便检索，但还是别人的东西

### 20-wiki/ — 知识层
消化后的系统化认知。每个子目录回答一个根本问题。
- 10-行业与市场："我在什么战场？"（行业全景、竞争格局、趋势、术语）
- 20-产品知识："我卖的东西好在哪？"（产品解析、竞品对比、FAQ、产品话术）
- 30-客户洞察："谁给我付钱、为什么？"（痛点地图、决策路径、异议处理、客户类型）
- 40-销售与转化："怎么成交？"（谈单方法论、话术库、报价策略、成交信号、转介绍）
- 50-内容与流量："怎么获取注意力？"（平台玩法、爆款拆解、标题公式、内容形式、发布时机）
- 60-实战复盘："我做过什么、学到了什么？"（成功案例、失败复盘、数据规律）

### 30-flow/ — 业务流程层
一条完整的生意流水线。只调用 wiki 的结论，不在这里写知识。
```
公域流量 → 私域运营 → CRM → 产品交付
  吸引       培育      转化     留存
```
- 10-公域流量：选题→创作→发布→数据
- 20-私域运营：朋友圈、社群、数据周报
- 30-CRM：线索→跟进→成交→售后（客户全生命周期）
- 40-产品交付：付费产品/服务的交付与维护

### 40-agents/ — AI能力层
用AI辅助的四个场景：
- 10-内容创作：文案、图文、视频脚本、选题生成
- 20-调研分析：资讯监控、竞品分析、行业研究
- 30-运营管理：CRM跟进、朋友圈运营、数据追踪
- 40-效率工具：文档生成、邮件处理、日程管理

### 50-projects/ — 实体层
跨流程的业务实体：
- 客户/：客户档案（不是CRM流程，是实体资料）
- 品牌/：品牌资产、视觉规范
- 账号/：各平台账号的详细资料

### 60-assets/ — 可复用资产
已经验证过、可以直接拿来用的成品素材：
- 内容模板/：各平台文章模板
- PPT模板/：用于培训/提案的PPT
- 报价方案/：给客户看的报价单模板
- 合同/：合同模板

### 99-archive/ — 归档
彻底结束的内容、过时的方案、关闭的项目。vault不会无限膨胀。

## 00-me 和 wiki 的分界线

| | 00-me（决策/选择题） | wiki（认知/简答题） |
|---|---|---|
| 客户 | "我选择服务谁" | "我理解他们什么" |
| 产品 | "我卖什么、卖多少钱" | "产品好在哪、跟别人比怎么样" |
| 风格 | "我的语气和调性" | "什么样的内容/话术经受过验证" |
| 账号 | "我有哪些号" | "各平台怎么玩" |

简单记：00-me 是选择题答案，wiki 是简答题答案。

## 引导填写设计原则

每个需要用户填写的文件都包含三部分：

```markdown
# [文件标题]

## 📌 填写指南
<!-- 告诉用户这个文件是干什么的、为什么要填、怎么填 -->

## ✏️ 你的内容
<!-- 用户实际填写的区域（填空式） -->

## 📖 参考：[行业]示例
<!-- 提供一个具体行业的完整示例，让用户有参照 -->
```

## 文件生命周期

文件从生到死不搬家，状态变更只改 frontmatter + 文件名前缀。

### 状态流转
```
📋选题中 → 📝草稿 → 📸拍摄中(仅视频) → ✂️剪辑中(仅视频) → 📤待发布 → ✅已发布 → 🚫Closed
```

### 双标记机制
1. 文件名前缀：每个内容文件以对应状态emoji开头
2. frontmatter status字段：在frontmatter中记录 status
